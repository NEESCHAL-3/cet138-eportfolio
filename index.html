1. What Is Full Stack Development?

Explanation

Full stack development means working with both the front end and the back end of a web application. The front end is what the user sees in the browser. The back end runs on a server, processes requests, applies rules, and connects to a database.

A full stack developer does not only make pages look good. They also need to understand how data moves through the application, how users interact with forms, how the server checks information, and how stored data is returned to the user safely.

For example, a login form on the front end can send an HTTP request to a Node.js and Express server. The server may check the login details against a MySQL or MongoDB database, then return JSON through an API so the browser knows whether the user has logged in successfully.

A simple API response might send back a small JSON object such as { "loggedIn": true }. This shows how the browser, server, application logic, and database work together in one full stack flow.

Reflection: this topic is important because the front end and back end must work together. If one layer is weak, the whole application can become slow, confusing, or insecure.

My learning note I understand full stack development better when I think of it as a request travelling through different layers, not as separate unrelated topics.

Simple request flow

1. Browser User fills in a form or clicks a button.
2. Front end HTML, CSS, and JavaScript display the page and collect input.
3. Back end Server code validates the request and runs the application logic.
4. Database Data is stored, updated, deleted, or returned to the server.
Example API response { "loggedIn": true, "message": "Login accepted" }

2. HTML

Explanation

HTML stands for HyperText Markup Language. It gives structure and meaning to web content. Headings, paragraphs, links, images, forms, lists, tables, and sections are all described using HTML elements.

I have used semantic HTML in this page. Semantic elements such as header, main, section, article, and footer make the page easier for browsers, screen readers, and developers to understand.

  • Headings create a clear page outline.
  • Alternative text explains images or visual placeholders.
  • Labels and form controls improve accessibility.
My learning note I used semantic tags in this example because they make the structure clearer than using only generic div elements.

3. CSS

Explanation

CSS stands for Cascading Style Sheets. It controls how HTML looks on the screen, including colours, spacing, borders, typography, layout, and hover effects. CSS also helps pages adapt to mobile, tablet, and desktop screens.

The example uses CSS Grid for layout, a CSS custom property for colour, and a media query so the cards stack on smaller screens. The colour picker changes a CSS variable with JavaScript, which shows that CSS and JavaScript can work together.

  • Grid: places the cards into columns.
  • Hover: gives feedback when the user points at a card.
  • Media query: changes the layout on small screens.
My learning note I kept the CSS example small so it is clear which rules control layout, colour, and interaction.

4. Bootstrap Framework

Explanation

Bootstrap is a front-end framework. It gives developers ready-made classes for layout, spacing, forms, buttons, cards, alerts, navigation, and other common interface parts. This can make development faster and helps keep the design consistent.

In this example I used Bootstrap's grid, card, badge, alert, button, and modal components. The modal opens using Bootstrap's JavaScript bundle, which shows that Bootstrap includes both CSS and interactive components.

  • .container keeps content aligned.
  • .row and .col-md-6 create a responsive grid.
  • .card, .badge, and .alert provide reusable components.
My learning note Bootstrap helped me see how a framework can speed up common layout and component work while still using normal HTML.

5. JavaScript

Explanation

JavaScript is used to add behaviour to a web page. It can respond to clicks, validate form input, change HTML content, update CSS values, and communicate with APIs. This makes a page dynamic instead of only static.

JavaScript can also use fetch() to send and receive data from APIs without reloading the whole page. For example, a page can ask a server for JSON data and update only the part of the page that needs to change.

The example is a small checklist. When the form is submitted, JavaScript stops the page from reloading, checks the input, creates a task object, stores it in an array, and updates the page. The complete and delete buttons change the same array and render the list again.

  • Event listener: reacts when the form is submitted.
  • Array: stores the current tasks.
  • DOM update: creates list items and buttons.
  • Validation: prevents empty tasks being added.
My learning note This example helped me understand that JavaScript changes the page by changing data first, then updating the DOM to match that data.
